1. Purpose
The purpose of this document is to define the business requirements for the Forensic Graph Intelligence Proof of Concept (Phase 1). This solution automates the extraction, linking, and visualization of entities and relationships from forensic investigation documents (PDF, DOCX, TXT) using LLM-powered AI.
The tool aims to significantly reduce manual effort in building intelligence graphs for fraud, anti-money laundering (AML), financial crime, and asset tracing investigations. It transforms unstructured documents into a structured, interactive, and queryable knowledge graph with AI-scored confidence and full source traceability.


2. Business Context
Forensic investigators currently rely on manual processes to read investigation documents, identify entities (persons, organizations, bank accounts, transactions, addresses), and construct link analysis charts using tools such as Microsoft Excel, i2 Analyst’s Notebook, or Microsoft Visio. This approach is time-consuming, error-prone, and difficult to scale when cross-referencing large document sets spanning multiple jurisdictions.
The Forensic Graph Intelligence PoC addresses these challenges by introducing an AI-powered pipeline that automatically extracts, resolves, and validates entities and their relationships from uploaded documents, and renders them as an interactive intelligence graph.

Pain Points
Manual entity extraction from investigation documents is slow and inconsistent across analysts.
No automated cross-document entity resolution or deduplication capability exists today.
Confidence and reliability of extracted intelligence is not scored or traceable to its source document.
No centralized way to query or explore the entity graph interactively.
Reporting outputs require manual assembly, increasing delivery time and inconsistency.
Path analysis between entities (e.g., tracing money flows) is performed manually.

Anticipated Impact
Reduce document-to-graph generation time from days to minutes.
Provide AI-scored confidence (1–10) with source traceability for every entity and relationship.
Enable interactive 2D and 3D graph exploration with automated path analysis.
Generate automated report-card style investigation summaries.
Strengthen EY’s forensic offering through a modern, intelligence-driven product.


Target Users
EY Forensic & Integrity Services investigators, analysts, and engagement leads (internal use).

Market for the Product
Internal EY forensic engagements across fraud, AML, financial crime, and asset tracing investigations. Initially targeting MENA region engagements with potential for global expansion given value has been proven.

3. Business Objectives
Automate entity and relationship extraction from unstructured investigation documents using Large Language Models (LLMs).
Provide a 3-pass AI pipeline (Extract → Cross-Reference → Validate) with confidence scoring and source attribution.
Deliver interactive 2D visualization of the intelligence network.
Enable path analysis between entities (shortest path, all paths) for tracing connections.
Support an analyst review workflow for low-confidence AI extractions (accept/reject).
Generate automated report-card style investigation summaries exportable as HTML/PDF.
Provide an AI chat interface for natural language querying of the graph.
Demonstrate AI Hub impact through a visible, high-value forensic use case.

6. User Experience
Users interact with the Forensic Graph Intelligence platform through the following components and workflows:
Right Sidebar — Configuration & Upload
Users select documents to upload, assign a document type per file, choose incremental or new graph mode, and trigger processing. 
Graph Canvas — 2D
The central area renders the intelligence graph. Users navigate 2D (vis.js), zoom, pan, and click nodes or edges to inspect details.
Left Panel — Node & Edge Detail
When a node or edge is selected, the left panel displays its full detail: name, type, confidence score with AI reasoning tooltip, evidence text, properties, and source document reference. Action buttons allow deletion, evidence update, and source document navigation.
Toolbar — Analysis & Manipulation
The horizontal toolbar provides search/filter, connect mode, add node, sticky notes, visual grouping, timeline view, path finder, undo, reset, and export controls.
Bottom Panel — 4 Tabs
A collapsible bottom panel hosts four tabs: AI Chat for natural language querying, Review Queue for triaging low-confidence items, Document Viewer for inline file preview, and Audit Log for viewing the full action history.
Export & Reporting
Users can export the graph as PNG/SVG/PDF images or generate a full HTML investigation report that includes entity and relationship tables, confidence summary, and audit log.

D. Architecture Overview
The Forensic Graph Intelligence PoC uses a lightweight, single-server architecture:
Backend: Python FastAPI application serving REST API endpoints.
Frontend: Single-page HTML/JavaScript application (Jinja2 template) with vis.js libraries.
LLM Integration: Azure OpenAI API via official Python SDKs.
Text Extraction: PyMuPDF for PDF, python-docx for DOCX, built-in file I/O for TXT/MD.
Storage: In-memory Python data structures (entities, relationships, undo stack). No persistent database.
Audit Log: JSONL file (audit_log.jsonl) for action history.

Acceptance Criteria
Text is extracted and chunked from all supported formats (.pdf, .docx, .doc, .txt, .md).
Document type is selectable per file from a predefined list of 9 types.
Unsupported file types are rejected with an informative error message.
Uploaded files are accessible in the Document Viewer tab.






Description:
The system shall process uploaded documents through a 3-pass AI pipeline: Pass 1 (Quick Extract) performs parallel chunk-level entity and relationship extraction using document-type-specific prompts. Pass 2 (Cross-Reference) resolves entity duplicates across documents and discovers missing relationships. Pass 3 (Validate & Score) assigns confidence scores (1–10) with reasoned explanations to each extraction. 

Acceptance Criteria:
Entities and relationships are extracted with confidence scores and source references.
Pipeline progress is visible to the user (current pass and item count).
Up to 5 concurrent LLM calls are processed in parallel during Pass 1.
Cross-document entity resolution produces merge mappings in Pass 2.
Each extraction includes a confidence score (1–10) and a human-readable reason in Pass 3.

Description:
The system shall deduplicate entities using fuzzy string matching (SequenceMatcher) with a configurable similarity threshold. After deduplication, relationship IDs are remapped and self-loops are removed. The resulting entities and relationships are stored in the in-memory graph.


Acceptance Criteria:
Duplicate entities of the same type with names exceeding the similarity threshold (default 0.70) are merged.
Merged entities retain the highest confidence score and combine evidence and source references.
Relationships are remapped to surviving entity IDs after merge.
Self-referencing relationships created by merges are removed.
Graph renders without orphan references.





Description:
The system shall render the extracted intelligence graph in both 2D (vis.js).  
Acceptance Criteria:
Graph displays all entities and relationships with correct colour coding and shapes.
Clicking a node opens the detail panel with name, type, confidence, evidence, properties, and source.
Clicking an edge opens the detail panel with relationship type, label, confidence, evidence, and source.
Confidence badge on detail panel shows a tooltip with the AI’s reasoning.

Acceptance Criteria:

Description:
The system shall provide a Path Finder that computes the shortest path (BFS) and all paths (DFS) between any two selected nodes, with configurable maximum depth.

Acceptance Criteria:
Shortest path is found using BFS with a configurable max depth (default 10).
All paths are found using DFS with a configurable max depth (default 5) and max 10 results.
Path results include the list of node IDs and edge IDs traversed.
Found paths can be highlighted on the graph.

Description:
The system shall provide an AI Chat interface in the bottom panel where analysts can ask natural language questions about the current graph. The system sends the full graph context (entities and relationships) to the LLM and returns concise, entity-citing answers.

Acceptance Criteria:
Chat accepts free-text questions and returns answers within the bottom panel.
Answers cite specific entity names and relationship types from the graph.
Chat interactions are recorded in the audit log.
Description:
The system shall automatically flag entities and relationships with a confidence score and present them in a Review Queue tab. Analysts can accept or reject (permanently moving the item to the rejected tab).

Acceptance Criteria:
Items with confidence score appear in the Review Queue, sorted by score ascending.
Rejecting an entity removes it and all its connected relationships from the graph
Rejecting a relationship removes only that relationship.
Rejecting moves an entity to the “Rejected” tab within the review flow.
Accept/reject actions are reflected in the graph immediately and logged in the audit trail.

Description:
The system shall generate an automated HTML investigation report containing an entity table, relationship table, confidence summary (high/medium/low counts), and the last 50 audit log entries. Additionally, the graph image can be exported as PNG, SVG, or PDF.

Acceptance Criteria:
HTML report contains all current graph data in tabular format.
Report includes a summary panel showing entity count, relationship count, and confidence breakdown.
Report is printable to PDF from the browser.
Graph image export supports PNG, SVG, and PDF formats.

Description:
The system shall log all user actions with a timestamp, action type, and relevant details. Actions include document uploads, node/edge creation and deletion, undo, reset, AI chat queries, review accept/reject, and path finder usage. The log is viewable in the Audit Log tab.

Acceptance Criteria:
Every create, delete, undo, upload, chat, review, and path-find action is logged.
